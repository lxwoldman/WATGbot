# **产品需求文档 (PRD)：全渠道供应链极速调度中台 (Supply Console Pro)**

## **1\. 项目背景与业务定位**

本项目旨在为跨境电商供应链调度员（Dispatcher）开发一套**极低延迟、全渠道聚合**的自动化交易控制台。

业务核心是处理海外供应商的高价值限时尾货报价单，赚取内部分销利润（点差）。由于行情瞬息万变，对海外报价的敏感度极高，系统必须彻底替代传统的人工切屏、复制粘贴和脑力算账，实现\*\*“毫秒级监控”**与**“一键式路由分发”\*\*。

## **2\. 核心角色与通信链路**

* **海外供应商 (Supplier)**：供货方，主要通过 **WhatsApp** 发送一手尾货报价。  
* **下游分销商 (Distributor)**：采购与接单方，主要通过 **Telegram** 接收报价指令。  
* **中控大脑 (System)**：即本系统，负责聚合双端消息、自动计算点差与库存分配、下发采购指令。  
* **【关键沟通形态】**：为了保证业务隐蔽性和沟通连贯性，**Telegram 端必须使用我司销售专员的真实个人账号（UserBot）直接与分销商私聊或在现有业务群内沟通，绝对禁止使用官方的 Telegram Bot (@xxx\_bot) 替代发送消息，因为下游 VIP 客户非常抗拒与机器人对话。**

## **3\. 技术栈规范 (强制要求及架构考量)**

为了确保底层库兼容性与全双工实时通信，本项目强制采用以下技术栈，请勿过度设计为沉重的微服务。

* **后端引擎：Node.js \+ Express**  
  * 【考量】：WhatsApp 扫码接入方案全在 JavaScript 生态中。Node.js 天生适合处理高并发的 I/O 密集型聊天事件。  
* **实时通信：Socket.io**  
  * 【考量】：确保前后端真正的“全双工长连接”，消息推送到控制台屏幕延迟需压缩在 10 毫秒以内。禁止使用 HTTP 轮询。  
* **WhatsApp 接入：whatsapp-web.js 或 Baileys**  
  * 【考量】：通过模拟网页端扫码登录我司业务账号，实现商业报价文本的自动化群发与监听。  
* **Telegram 接入 (核心定制)：基于 MTProto 的 Node 库（如 telegram (GramJS) 或 tdlib）**  
  * **【强制要求】**：必须实现 **UserBot (个人账号客户端模拟)**。  
  * 【考量】：不能使用官方 Bot Token 方案。系统需要像第三方 TG 客户端一样，通过请求发送手机验证码的方式接管销售的个人 TG 账号，以真人的名义收发消息，确保分销商无感。  
* **数据库与缓存：PostgreSQL \+ Redis**  
  * 【考量】：PostgreSQL 用于存储高一致性的通道配置映射；Redis 用于高速读取/更新各分销商当前的剩余采购额度状态。  
* **前端架构：原生 HTML/CSS/JS**  
  * 【考量】：使用已提供的原生高保真 UI 原型，仅注入 socket.io-client 绑定事件，避免使用 React/Vue 以追求极致渲染速度和缩短开发周期。

## **4\. 核心工作流 (Happy Path)**

1. **\[监听\]** WA 供应商发送备货单（包含：品牌 \+ 品类）。  
2. **\[抓取\]** Node.js 监听到新消息，通过 Socket 瞬间推送到前端 UI 的“供应商情报池”。  
3. **\[路由交互\]** 用户在前端点击快捷按钮 \[0\] 或 \[1\]，前端向后端发 Socket 指令，后端调用 WA 接口回复供应商已收到。  
4. **\[分发备货单\]** 用户在中控区点击 \[发预备单\]，后端调用 **TG UserBot API**，以销售个人号名义向所有打勾激活的分销商并发推送文本。  
5. **\[核心出货\]** 供应商下发具体商品报价（如 批次A @ 0.90）。前端自动提取原始报价 0.90，根据系统配置自动扣减我司利润点差（如 0.03），得出最终下发分销价 0.87。  
6. **\[分发报价\]** 用户点击 \[发报价\]，带上计算好的最终价格，通过 TG 个人号分发给下游。  
7. **\[回单结算\]** 分销商在 TG 确认数量完毕，用户在前端卡片点击 \[回执\]，系统自动拼接发货单文本，一键通过 TG 个人号发送到该客户专属窗口。

## **5\. 异常处理机制 (价格跳水熔断)**

* **业务痛点**：供应商给 0.90 报价，但市场行情突变，下游分销群反馈目前别家只要 0.88。必须在秒级内叫停并向供应商报备压价。  
* **【自动化处理流程】**：  
  1. 后端 TG UserBot 监听到分销商窗口出现纯数字或价格特征文本（如 0.88），立刻通过 Socket 推送前端触发红灯警报。  
  2. 用户只需盲点全屏最醒目的 \[一键反馈给供应商 0.88\] 按钮，系统立刻通过 WA 报备，同时自动通过 TG 向所有下游广播 等等/先暂停，将价格危机处理时间压缩至极点。

## **6\. 前后端交互模块拆解 (UI 映射)**

### **6.1 界面交互设计理念**

* **极度压缩视觉空间**：右侧分销商卡片采用极其紧凑网格布局，确保无垂直滚动条，余光掌控全局。  
* **读写分离与视觉降噪**：系统抓取信息无边框加粗显示，仅手输区域（如点差）保留输入框形态。  
* **高频操作一键化**：繁琐的订单文本拼接由 JS 自动完成，用户仅需点击单点指令按钮。

### **6.2 左侧：供应商情报池 (Source Pool)**

* **通道切换**：支持多供应商下拉切换。  
* **消息流与一键回复**：展示最新核心报价，点击快捷按钮即向**当前选中供应商**发信。

### **6.3 中间：交易控制中心 (Central Brain)**

* **数据提取与计算**：完成最终分销价的计算与采购额度缺口的报警逻辑。  
* **快捷广播矩阵**：遍历右侧激活状态的分销商卡片（获取对应的 TG User ID / Chat ID），通过 UserBot 并行执行发信任务。

### **6.4 右侧：分销调度与回执池 (Dispatcher)**

* **资源配置表**：包含分销商身份绑定（需映射后台的 TG Chat ID）、分配采购额度输入，以及针对单一渠道的 \[报价\], \[回执\], \[预备\] 指令按钮。  
* **专属回单生成器**：选定目标后自动拼接确认发货文本，通过 UserBot API 独立投递到客户聊天窗口。

## **7\. 给 AI 开发者的阶段实施建议 (Next Steps)**

请以此顺序逐步展开开发，避免陷入过度设计：

* **Phase 1: 基础管道建设与双平台登录**  
  * 初始化 Node \+ Express \+ Socket.io 项目。  
  * **难点攻克 (WA)**：集成 whatsapp-web.js，实现终端二维码输出与扫码登录，捕获测试消息。  
  * **难点攻克 (TG UserBot)**：集成 GramJS 等 MTProto 库，开发独立的终端脚本输入手机号+验证码完成销售账号的登录授权，并生成长期 session 凭证。  
* **Phase 2: UI 与后端的 Socket 握手**  
  * 将前端 HTML 的模拟按钮事件改造为真实的 socket.emit。  
  * 实现双平台收到的消息能实时推送到左侧面板和价格拦截器。  
* **Phase 3: 数据结构与路由调度**  
  * 设计配置体系（或简易 DB 表），映射“分销商1”与“真实的 TG Chat ID”。  
  * 打通中间件：UI 点击发报价 \-\> Socket 传参 \-\> Node.js 找到对应 ID \-\> 调用 WA/TG API 发信。  
* **Phase 4: 业务算法闭环**  
  * 联调点差计算逻辑、额度缺口校验、以及回单文本格式化的全流程。
