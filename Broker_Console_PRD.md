# **产品需求文档 (PRD)：全渠道极速交易中台 (Broker Console Pro)**

## **1\. 项目背景与业务定位**

本项目旨在为足球博彩经纪人（Broker）开发一套**极低延迟、全渠道聚合**的自动化交易控制台。

业务核心是处理海外机构的高价值亚盘指令（蛇货），赚取水差（退点）。由于市场瞬息万变，对盘口水位的敏感度极高，系统必须彻底替代传统的人工切屏、复制粘贴和脑力算账，实现\*\*“毫秒级监控”**与**“一键式路由分发”\*\*。

## **2\. 核心角色与通信链路**

* **源头 (Source)**：发单方，主要通过 **WhatsApp**（部分使用 Telegram）发送盘口。  
* **资源 (Resource)**：接单下注方，主要通过 **Telegram**（部分使用 WhatsApp）接收指令。  
* **中控大脑 (Broker/System)**：即本系统，负责聚合双端消息、自动计算退点与额度分配、下发指令。

## **3\. 技术栈规范 (强制要求及架构考量)**

为了确保底层库兼容性与全双工实时通信，本项目强制采用以下技术栈，请勿过度设计为沉重的微服务。以下是每项技术选型的核心原因与预期效果：

* **后端引擎：Node.js \+ Express**  
  * **【为什么这么做】**：目前全网最成熟、稳定的非官方 WhatsApp 扫码接入方案全都在 JavaScript 生态中。如果强行使用 Go/Java 会导致严重的跨语言调用灾难。Node.js 天生适合处理高并发的 I/O 密集型聊天事件。  
  * **【最终效果】**：开发速度极快，生态无缝衔接，能够轻松同时挂载数百个并发的聊天监听而不卡顿。  
* **实时通信：Socket.io**  
  * **【为什么这么做】**：传统的 HTTP 轮询（Polling）会有几秒的延迟，在高频交易中是致命的。我们需要服务器一旦收到盘口，无需用户刷新，立刻推送到前端。  
  * **【最终效果】**：实现前后端真正的“全双工长连接”，消息从源头发出到显示在控制台屏幕上，延迟压缩在 10 毫秒以内。  
* **WhatsApp/Telegram 接入协议**  
  * **【为什么这么做】**：业务性质决定了无法使用官方审核极严的 WhatsApp Cloud API。必须使用 whatsapp-web.js (模拟无头浏览器扫码登录) 和 TG 官方 Bot Token。  
  * **【最终效果】**：绕过官方风控限制，实现任意博彩话术文本的自动化群发，且支持多个账号矩阵同时在线。  
* **数据库与缓存：PostgreSQL \+ Redis**  
  * **【为什么这么做】**：财务对账一分不能错，所以使用高一致性的 PostgreSQL 存储静态配置（如 Channel ID 映射）；而各资源当前的剩余额度变化极快，每次都读写数据库太慢，需使用 Redis。  
  * **【最终效果】**：既保障了资金结算的绝对准确，又保证了额度扣减计算的高速读取。  
* **前端架构：原生 HTML/CSS/JS**  
  * **【为什么这么做】**：避免使用 React/Vue 等重型虚拟 DOM 框架。对于单一页面的交易终端，框架本身的生命周期和状态更新会导致微小的渲染延迟。  
  * **【最终效果】**：极其轻量、渲染速度达到极致，且已完成高保真 UI 原型，后端工程师只需专注对接 Socket 事件，极大缩短开发周期。

## **4\. 核心工作流 (Happy Path)**

1. **\[监听\]** WA 源头发送预备单（联赛+队伍）。  
2. **\[抓取\]** Node.js 监听到新消息，通过 Socket.io 瞬间推送到前端 UI 的“源头情报池”。  
3. **\[路由交互\]** 用户在前端点击快捷按钮 \[0\] 或 \[1\]，前端向后端发 Socket 指令，后端调用 WA 接口回复源头。  
4. **\[分发预备\]** 用户在中控区点击 \[发预备单\]，后端向所有打勾激活的 TG 资源群并发推送文本。  
5. **\[核心出货\]** 源头下发具体盘口（如 大2 @ 0.90）。前端自动提取水位 0.90，根据系统配置自动扣减退水（如 0.03），得出最终发送水位 0.87。  
6. **\[分发盘口\]** 用户点击 \[发盘口\]，带上计算好的最终水位，分发给资源。  
7. **\[回单结算\]** 资源在 TG 报单完毕，用户在前端资源卡片点击 \[回执\]，系统自动提取该资源的交货金额，按标准格式拼接回单文本，一键通过 TG Bot 发送到该资源专属群。

## **5\. 异常处理机制 (掉水熔断)**

* **业务痛点**：源头给 0.90，但资源群由于行情变化反馈当前只有 0.88。如果人工去输入 0.88 并切换软件发送，需要 3-5 秒，此时 0.88 可能也没有了。  
* **【为什么这么做】（自动化拦截的设计初衷）**：  
  将“发现掉水 \-\> 人工输入 \-\> 切换软件 \-\> 报告”的四步人工操作，压缩为“系统正则监控 \-\> 弹窗报警 \-\> 点击一键反馈”的两步半自动操作。  
* **【最终效果】**：  
  1. 后端 TG Bot 监听到纯数字或包含水位的文本（如 0.88），立刻通过 Socket 推送给前端触发红灯警报。  
  2. 用户**无需打字**，只需盲点全屏最醒目的 \[一键反馈给源头 0.88\] 按钮，系统立刻通过 WA 报备，整个危机处理时间从 5 秒缩短至 0.5 秒。

## **6\. 前后端交互模块拆解 (UI 映射与 UX 哲学)**

### **6.1 界面交互设计理念 (UI/UX 哲学)**

* **【极度压缩的视觉空间】**：右侧资源卡片采用极其紧凑的网格布局（去除多余空白和上下边距），**确保所有资源和底部回单区能在一屏内完全展示，绝对禁止出现垂直滚动条**。  
  * **效果**：用户盯盘时视线不用上下移动，余光就能掌控所有资源的状态。  
* **【读写分离与视觉降噪】**：系统自动抓取的比赛队伍、联赛信息采用**无边框加大加粗文本**；仅有需要手动输入的地方（退点、额度）保留输入框。  
  * **效果**：极大降低用户的认知负担，消除“我在填表”的压迫感，凸显核心数据。  
* **【高频操作一键化】**：将所有繁琐的文本拼接（如：1.韩K联 江原v安阳 大2 @ 0.87确5000）全部交给 JS 自动生成，暴露给用户的只有 \[盘口\]、\[回执\] 等单点按钮。

### **6.2 左侧：源头情报池 (Source Pool)**

* **通道切换**：支持多个 WA/TG 源头的下拉切换，切换后聊天记录与上下文自动变更。  
* **消息流**：仅展示最新的核心指令。  
* **一键回复**：点击 0/1/2 按钮，向**当前选中的源头通道 ID** 发送对应文本。

### **6.3 中间：交易控制中心 (Central Brain)**

* **数据提取与计算**：  
  * 原始水位 (从消息提取) \- 扣除退点 (预设) \= 最终发送水位 (自动计算并锁定)。  
  * 总交货目标 \- 资源已分配额度 (各资源输入框之和) \= 未分配缺口 (大于 0 标红报警)。  
* **快捷广播矩阵**：\[发预备\], \[发盘口\], \[发等等\], \[发取消\] 等。点击后，遍历右侧所有处于“激活”状态的资源卡片，取出它们的绑定的 Channel ID，并行执行发信任务。

### **6.4 右侧：资源调度与回执池 (Resource Dispatcher)**

* **资源配置表 (动态生成)**：每行代表一个出货资源。  
  * **身份绑定**：必须包含后台数据库对应的 Channel\_ID（如 TG:-100123456）。  
  * **单点操作**：\[盘口\], \[回执\], \[预备\]。点击仅对当前一行的 Channel\_ID 发送单边指令。  
* **专属回单生成器 (位于右下角不遮挡区)**：  
  * 选定目标资源后，自动拼接确认文本。点击发送，通过 API 独立投递到该资源的聊天窗口。

## **7\. 给 AI 开发者的阶段实施建议 (Next Steps)**

请以此顺序逐步展开开发，避免代码冗长导致错误：

* **Phase 1: 基础管道建设**  
  * 初始化 Node \+ Express 项目，跑通 Socket.io 服务端与客户端（前端 HTML）的握手。  
* **Phase 2: 协议层接入**  
  * 引入 whatsapp-web.js，实现终端输出二维码 \-\> 手机扫码 \-\> 成功捕获第一条 WA 消息并打印。  
  * 引入 node-telegram-bot-api，配置 Token，实现在特定群组的收发测试。  
* **Phase 3: 数据结构与路由调度**  
  * 设计数据库的 Channels 表，区分不同聊天窗口的 ID 映射。  
  * 将前端写死的 UI 数据（联赛、盘口）替换为通过 Socket 接收的动态变量。  
* **Phase 4: 业务算法注入**  
  * 在前后端完善退点算水逻辑、额度配资校验、以及回单文本格式化引擎。
